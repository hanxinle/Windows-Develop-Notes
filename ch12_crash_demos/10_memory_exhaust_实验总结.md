# 10_memory_exhaust.cpp 实验总结

## 1. 实验目的

理解 C++ 程序中 **内存分配失败** 和 **内存泄漏** 两类问题的表现形式、危害以及正确的处理策略。

本 demo 演示：

- 超大内存请求失败时如何检查返回值。
- 内存不足时如何通过 **优雅降级（Graceful Degradation）** 避免崩溃。
- 内存泄漏如何缓慢累积并最终拖垮进程或系统。

这些都不是“立即崩溃”的 bug，而是 **运行中逐渐恶化、最终诱发崩溃或 OOM 的慢性问题**，因此是 crash 治理中不可忽视的一类。

## 2. 实验环境

- Visual Studio 2022
- x64 Debug / x64 Release
- 编译选项：`/EHsc`、`/W4`、C++17
- 依赖库：无（仅需 `<windows.h>`、`<cstdio>`、`<vector>`）

> 注意：测试 2（`TestExhaustMemory`）默认被注释掉，因为它会不断分配内存直到失败，可能长时间占用大量物理内存和页面文件，影响系统稳定性。如需观察，请在 `main()` 中取消注释并谨慎运行。

### 3.2 测试 1：超大内存分配失败

```text

  尝试分配 100 GB...
  分配失败，返回 nullptr.
```

请求 100 GB 内存时，`malloc` 无法找到足够大的连续地址空间，返回 `nullptr`。这是典型的 **一次性大内存请求失败**。

### 3.3 测试 3：优雅降级

```text
[测试 3] 优雅降级示例...

  实际缓存大小: 500.00 MB (期望值 500.00 MB)
```

程序希望分配 500 MB 缓存，若失败则逐步减半（250 MB、125 MB……直到 1 MB），最终成功分配到一个可用大小。这就是 **优雅降级**：功能可能受限，但程序不会崩溃。

### 3.4 测试 4：内存泄漏

```text
[测试 4] 内存泄漏模拟...
  分配内存但不释放，模拟泄漏.
  泄漏了 100 MB 内存（这些内存已无法回收）.
```

100 次 1 MB 的分配都没有 `free`，模拟了内存泄漏。每次泄漏后程序仍继续运行，但可用内存逐渐减少。

## 4. 核心结论

### 4.1 `malloc` / `new` 可能返回 `nullptr`

C++ 中动态内存分配不是一定能成功的。当请求过大、系统内存不足、地址空间碎片化或进程配额受限时，分配函数会失败：

```cpp
void* p = malloc(hugeSize);
if (p == nullptr) {
    // 分配失败，必须处理
}
```

> 注意：`new` 默认在失败时抛出 `std::bad_alloc`，除非使用 `new (std::nothrow)`。

### 4.2 内存不足时的处理策略

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| **检查返回值** | 所有动态分配 | 大内存分配后判断 `nullptr` |
| **优雅降级** | 非核心功能 | 缓存大小减半、关闭特效、降低精度 |
| **重试/等待** | 临时性内存紧张 | 短暂等待后再次尝试 |
| **安全退出** | 核心资源无法满足 | 保存状态后退出，而不是崩溃 |
| **限制上限** | 长期运行服务 | 设置内存使用阈值，超过则告警或限流 |

本 demo 的测试 3 就是优雅降级的典型实现：

```cpp
size_t actualCacheSize = desiredCacheSize;
while (actualCacheSize >= 1024 * 1024) {
    cache = malloc(actualCacheSize);
    if (cache != nullptr) break;
    actualCacheSize /= 2;  // 逐步减半
}
```

### 4.3 内存泄漏不会立即崩溃，但会间接诱发崩溃

内存泄漏的特点是：

- **隐蔽**：每次泄漏一点点，程序继续正常运行。
- **累积**：长时间运行后，泄漏总量可能达到 GB 级别。
- **后果严重**：最终触发 OOM（Out of Memory），进程被系统终止；或导致其他分配失败，引发连锁崩溃。

在 crash 治理中，内存泄漏属于 **间接崩溃原因**：崩溃点可能在最后那次分配失败，但根因是之前某段代码没有释放内存。

### 4.4 测试 2 的作用与风险

`TestExhaustMemory` 演示的是 **逐步耗尽系统内存** 的过程：

```cpp
while (true) {
    void* p = malloc(blockSize);
    if (p == nullptr) break;
    memset(p, 0xAB, blockSize);  // 写入数据，确保内存真正提交
    blocks.push_back(p);
}
```

它会持续分配 10 MB 块，直到 `malloc` 返回 `nullptr`。这个测试默认被注释掉，原因是：

- 可能分配数 GB 甚至更多内存。
- 会触发系统大量换页，导致整机卡顿。
- 在内存较小的机器上可能影响其他进程。

如需运行，建议在虚拟机或内存充足的机器上操作，并准备好任务管理器观察内存曲线。

## 5. 深入分析

### 5.1 为什么内存耗尽会被纳入 crash 治理

`crash_blog.md` 把内存相关问题列为常见 crash 类型，原因如下：

| 原因 | 说明 |
|------|------|
| **直接终止进程** | 系统 OOM killer 或 CRT 检测到无法分配时，可能直接终止进程。 |
| **诱发空指针解引用** | 分配失败后如果代码继续假设指针有效，会访问 `nullptr`，触发访问冲突。 |
| **堆破坏风险** | 内存紧张时，某些代码可能错误地复用已释放内存，造成 use-after-free 或双重释放。 |
| **性能雪崩** | 物理内存耗尽后大量换页，程序响应变慢，可能触发超时、 watchdog 重启或用户强制结束。 |
| **dump 定位困难** | 崩溃点往往在最后那次分配或访问，真正原因却是远处持续泄漏。 |

因此，内存治理的目标不仅是“不泄漏”，还包括 **失败时可控、不足时可降级**。

### 5.2 32 位程序的地址空间限制

32 位进程即使运行在 64 位 Windows 上，通常也只有 **2 GB 用户态地址空间**（开启 `/LARGEADDRESSAWARE` 后最多 3 GB 或 4 GB）。这意味着：

- 32 位程序更容易因地址空间耗尽而分配失败，即使物理内存还有很多。
- 64 位程序拥有近乎无限的虚拟地址空间，但仍受物理内存和提交限制约束。

工程建议：对于需要处理大内存的应用，优先编译为 64 位。

### 5.3 提交（Commit）与保留（Reserve）

Windows 虚拟内存管理有两个概念：

- **Reserve**：预留一段虚拟地址范围，不实际消耗物理内存或页面文件。
- **Commit**：把预留的地址真正提交给系统，分配物理内存或页面文件。

本 demo 中的 `malloc` 通常直接完成 commit。如果调用 `VirtualAlloc` 并只 reserve 不 commit，即使请求很大也可能成功，但首次访问时会触发 commit，可能失败。

### 5.4 Windows 的内存限制机制

除了物理内存不足，以下因素也会导致分配失败：

- **进程工作集限制**：通过 `SetProcessWorkingSetSize` 或作业对象限制。
- **页面文件不足**：系统提交总量超过页面文件大小时，`VirtualAlloc` 会失败。
- **地址空间碎片化**：长时间运行后，即使总空闲内存足够，也可能没有足够大的连续区域。
- **堆碎片化**：CRT 堆经过多次分配释放后，可能出现很多不连续的小块。

### 5.5 内存泄漏检测工具

| 工具 | 用途 |
|------|------|
| AddressSanitizer (ASan) | 运行时检测泄漏、越界、use-after-free |
| Visual Studio CRT 泄漏检测 | `_CrtDumpMemoryLeaks`、`_CrtSetDbgFlag` |
| Application Verifier | Windows 官方，检测堆错误和泄漏 |
| Windbg `!heap -stat` | 分析 dump 中的堆使用情况 |
| VLD (Visual Leak Detector) | Visual Studio 插件，定位泄漏调用栈 |

## 6. 工程启示

1. **所有动态分配都要检查返回值**。`malloc` 返回 `nullptr`、`new` 可能抛异常，都不要假设一定成功。
2. **大内存分配前评估合理性**。不要一次性请求远超系统容量的内存。
3. **内存不足时优雅降级**。缓存、特效、精度等非核心功能可以缩减或关闭。
4. **避免内存泄漏**。每个 `new` 有 `delete`，每个 `malloc` 有 `free`；优先使用智能指针。
5. **使用 RAII 管理资源**。`std::unique_ptr`、`std::shared_ptr`、`std::vector` 自动释放，减少遗漏。
6. **开发阶段启用 ASan**。在编译选项中加入 `/fsanitize=address`，能自动发现泄漏和越界。
7. **监控进程内存**。对长期运行的服务，设置内存使用阈值，超过则告警或重启。
8. **32 位应用注意地址空间**。大内存场景应迁移到 64 位。
