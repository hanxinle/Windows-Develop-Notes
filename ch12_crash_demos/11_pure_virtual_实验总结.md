# 11_pure_virtual.cpp 实验总结

## 1. 实验目的

理解 C++ 中 **纯虚函数调用（Pure Virtual Function Call）** 这一类运行时错误的触发场景，重点掌握：

- 构造函数和析构函数中调用虚函数的特殊规则。
- 对象已析构后通过悬空指针调用虚函数的未定义行为。
- 如何使用智能指针和 RAII 避免生命周期相关的虚函数调用错误。

这类错误通常表现为运行时错误 **R6025 - pure virtual function call**，是 Windows C++ 程序中相对隐蔽但定位明确的崩溃类型。

## 2. 实验环境

- Visual Studio 2022
- x64 Debug / x64 Release
- 编译选项：`/EHsc`、`/W4`、C++17
- 依赖库：无（仅需 `<cstdio>`、`<memory>`）

析构 `DerivedWithDtorCall` 时，先执行派生类析构函数，再执行基类析构函数。在基类析构函数期间，对象的动态类型已回退为 `BaseWithDtorCall`，所以调用的是基类版本。

## 4. 核心结论

### 4.1 C++ 对象构造和析构期间的动态类型

C++ 中对象的动态类型在构造和析构期间是**逐步变化**的：

| 阶段 | 对象动态类型 | 虚函数调用目标 |
|------|------------|--------------|
| 基类构造函数执行期间 | 基类 | 基类版本 |
| 派生类构造函数执行期间 | 派生类 | 派生类版本 |
| 对象完整构造后 | 派生类 | 派生类版本 |
| 派生类析构函数执行期间 | 派生类 | 派生类版本 |
| 基类析构函数执行期间 | 基类 | 基类版本 |

因此，在基类构造函数或析构函数中调用虚函数，不会调用到派生类的重写版本。

### 4.2 构造函数中直接调用纯虚函数通常会导致链接失败

如果在基类构造函数中直接调用本类的纯虚函数，例如：

```cpp
class Base {
public:
    virtual void DoWork() = 0;
    Base() { DoWork(); }  // 直接调用纯虚函数
};
```

MSVC 在链接阶段会报错：

```text
error LNK2019: 无法解析的外部符号 "public: virtual void __cdecl Base::DoWork(void)"
```

原因是：编译器把 `DoWork()` 当作普通虚函数调用处理，链接时需要找到 `Base::DoWork()` 的实现，但纯虚函数没有实现。

所以这种写法**连编译链接都通不过**，更到不了运行时的 R6025。这也说明 MSVC 能在早期就发现这种明显错误。

### 4.3 析构后调用虚函数是未定义行为

对象析构后，其内存可能被回收、复用或保持不变。通过悬空指针调用虚函数：

- 可能“碰巧正常”执行。
- 可能输出垃圾数据。
- 可能立即触发访问冲突崩溃。
- 可能被运行时检测为纯虚函数调用错误。

**“没有崩溃”不等于“代码正确”**，这是未定义行为的典型特征。

### 4.4 为什么 Release 模式下可能还能运行

你之前在 Release 模式下测试发现程序还能运行，这正是未定义行为的表现：

- Release 优化后，栈帧布局、寄存器使用和内存复用策略与 Debug 不同。
- 如果析构后的栈内存或堆内存恰好还没被覆盖，vptr 仍然指向原来的虚函数表，调用就会“成功”。
- 这只是一个巧合，换一批代码、换个编译器版本、换个运行环境，行为可能就完全不同。

因此，不能依赖“现在没崩溃”来判断代码安全。

## 5. 深入分析

### 5.1 为什么纯虚函数调用会被纳入 crash 治理

`crash_blog.md` 把纯虚函数调用相关问题纳入 crash 治理，原因如下：

| 原因 | 说明 |
|------|------|
| **运行时错误对话框** | R6025 会弹出系统错误框，直接终止进程，用户体验极差。 |
| **难以静态发现** | 代码在语法上是合法的，但 MSVC 对明显错误（如基类构造函数调用纯虚函数）会直接链接失败。 |
| **析构期间常见** | 基类析构函数中调用虚函数是很多成熟项目也曾踩过的坑。 |
| **多态回调场景** | 观察者模式、委托模式中，对象销毁后未清理回调，极易触发。 |
| **dump 中有明确签名** | R6025 的崩溃栈通常能直接定位到虚函数调用点，修复相对直接。 |

### 5.2 虚函数表的内存布局

C++ 对象内存中通常包含一个指向虚函数表（vtable）的指针（vptr）。

- 构造时：先设置基类的 vptr 指向基类 vtable，再设置派生类的 vptr 指向派生类 vtable。
- 析构时：先把 vptr 改回基类 vtable，再执行基类析构函数。

所以在基类构造函数和析构函数执行期间，vptr 都是基类的，调用虚函数自然只能调用基类版本。

### 5.3 悬空指针与内存覆盖

测试 2 中：

```cpp
Base* ptr = nullptr;
{
    Derived d;
    ptr = &d;
}
ptr->DoWork();  // d 已析构，ptr 悬空
```

`d` 在栈上，析构后栈内存仍然保留，vptr 可能还没被覆盖，所以调用可能成功。但如果是堆对象：

```cpp
Base* ptr = new Derived();
delete ptr;
ptr->DoWork();  // 堆内存已释放，更可能崩溃
```

堆内存释放后可能被 CRT 回收或设置哨兵，调用崩溃的概率更高。但无论哪种情况，都是未定义行为。

### 5.4 基类构造函数中直接调用纯虚函数会发生什么

有些开发者以为下面这段代码能触发 R6025：

```cpp
class Base {
public:
    virtual void DoWork() = 0;
    Base() { DoWork(); }
};

class Derived : public Base {
public:
    void DoWork() override { printf("Derived::DoWork()\n"); }
};

int main() {
    Derived d;
    return 0;
}
```

但实际上 MSVC 会在链接阶段报错：

```text
error LNK2019: 无法解析的外部符号 "public: virtual void __cdecl Base::DoWork(void)"
```

原因是编译器在 `Base::Base()` 中生成了对 `Base::DoWork()` 的调用，但 `Base::DoWork()` 是纯虚函数，没有实现，链接器找不到符号。

这说明：**基类构造函数中直接调用本类纯虚函数，在 MSVC 下是链接错误，不是运行时 R6025**。

R6025 通常出现在更隐蔽的场景，例如：

- 通过间接方式（如函数指针、已释放对象的 vptr）调用到纯虚函数。
- 对象已析构后，悬空指针恰好指向了被改写的 vtable 区域。
- 某些第三方库或 CRT 内部路径触发了非法虚函数派发。

这类情况难以稳定复现，所以本 demo 没有默认启用，仅作为概念说明。

## 6. 工程启示

1. **构造函数和析构函数中避免调用虚函数**。如果必须调用，明确调用当前类的非虚版本或具体实现。
2. **不要保留指向已析构对象的指针**。删除对象后立即把指针置为 `nullptr`，或使用智能指针。
3. **使用智能指针管理生命周期**。`std::unique_ptr` / `std::shared_ptr` 能自动释放对象，避免悬空指针。
4. **观察者模式中注意注销回调**。对象销毁前，从观察者列表中移除自己。
5. **不要依赖“没崩溃就是正确”**。未定义行为在 Debug/Release、不同机器上表现可能完全不同。
6. **关注编译器警告**。MSVC 对构造函数中调用虚函数通常会给出 C4526、C4584 等警告，不要忽略。
7. **崩溃 dump 中出现 R6025 时重点检查生命周期**。通常是对象已销毁后仍被使用，或构造/析构期间误调虚函数。

## 7. 编译问题回顾

本 demo 编译相对简单，但需要注意：

- 若源码文件保存为 UTF-8 无 BOM，MSVC 可能报 `warning C4819` 或编码相关错误。解决方法是保存为 **UTF-8 with BOM**，或在项目属性中添加 `/utf-8` 编译选项。
- 测试 2 中的危险调用默认被注释，以确保程序能完整运行所有测试。
