# 07_race_condition.cpp 实验总结

## 1. 实验目的

理解 C++ 多线程程序中 **数据竞争（Data Race）** 的产生原因、表现形式和危害，掌握三种常见防护手段：

- `std::mutex` / `std::lock_guard` 互斥锁。
- `std::atomic` 原子操作。
- 避免对共享可变数据的无保护并发访问。

同时体会为什么“多线程 bug 在本地难以复现、上线后偶发”是 C++ 工程中的经典难题。

## 2. 实验环境

- Visual Studio 2022
- x64 Debug / x64 Release
- 编译选项：`/EHsc`、`/W4`、C++17（`07_race_condition.cpp` 未使用 C++17 特有语法，但项目统一开启）
- 依赖库：无（仅需 `<windows.h>`、`<thread>`、`<mutex>`、`<atomic>`、`<vector>`、`<cstdio>`）

> 注意：本次实验遇到了 `warning C4819` 及编译错误，原因是源文件为 UTF-8 无 BOM 格式，而 MSVC 默认按 GBK（代码页 936）读取。已将文件保存为 **UTF-8 with BOM** 后解决。

## 3. 运行结果

### 3.1 测试 1：无锁竞争

```text
[测试 1] 无锁多线程竞争...
  10 个线程各增加 10000 次，预期结果 = 100000.
  预期值: 100000
  实际值: 92974
  结果: 错误（发生竞争）
```

多个线程并发执行 `g_counter++`，实际值通常小于 `100000`。本次运行得到 `92974`，具体数值每次运行都可能不同。

```text
  结果: 正确
```

加锁后结果始终正确。

```text
[测试 3] 链表竞争（可能丢失节点或内存泄漏）...
  多个线程同时往链表头部插入节点，不加锁.
  预期节点数: 20000
  实际节点数: 19990
  结果: 错误（发生竞争，部分节点丢失）
```

无保护并发插入链表，部分节点被覆盖或丢失。


原子自增无需加锁即可保证结果正确。

## 4. 核心结论

### 4.1 数据竞争是未定义行为

C++ 标准规定：当多个线程同时访问同一个内存位置，且至少有一个是写操作，且没有同步机制时，即构成 **数据竞争（Data Race）**，属于未定义行为（Undefined Behavior）。

未定义行为意味着：

- 结果可能正确，也可能错误。
- 程序可能崩溃，也可能不崩溃。
- 同一份代码在不同 CPU、不同编译器、不同运行次数下表现可能完全不同。

### 4.2 `g_counter++` 不是原子操作

虽然 `g_counter++` 在源码中只是一行代码，但编译后的机器指令通常分为三步：

1. 从内存读取 `g_counter` 到寄存器。
2. 寄存器值加 1。
3. 将寄存器值写回内存。

如果线程 A 执行完步骤 1 后，线程 B 也读取了旧值，那么两个线程都会基于同一个旧值加 1，最终只增加了 1 而不是 2。这就是测试 1 中实际值小于预期值的原因。

### 4.3 `std::lock_guard` 提供互斥访问

```cpp
std::lock_guard<std::mutex> lock(g_mutex);
g_counterLocked++;
```

`std::lock_guard` 在构造时加锁，在析构时自动解锁（RAII）。它可以确保同一时刻只有一个线程进入临界区，从而避免数据竞争。

### 4.4 `std::atomic` 适用于简单原子操作

```cpp
std::atomic<int> g_atomicCounter{0};
g_atomicCounter++;
```

对于计数器、标志位等简单场景，`std::atomic` 比互斥锁更高效，因为它直接利用 CPU 提供的原子指令（如 x86 的 `LOCK INC`），不需要进入内核态。

### 4.5 复杂数据结构必须整体加锁

测试 3 中的链表插入涉及多步操作：

```cpp
node->next = g_head;  // 步骤 1
g_head = node;        // 步骤 2
```

这两步之间如果发生线程切换，另一个线程可能读到不一致的状态，导致节点丢失或链表断裂。对于这类复合操作，不能依赖原子变量，必须使用互斥锁保护整个操作序列。

### 4.6 为什么多线程竞争会被纳入 crash 治理

`crash_blog.md` 把"多线程竞争"列为 14 种常见 crash 类型中的第 4 种，原因如下：

| 原因 | 说明 |
|------|------|
| **破坏内存结构** | 无保护并发写链表、树、哈希表等结构时，指针关系会被打乱，造成节点丢失或链断裂。 |
| **诱发其他崩溃** | 竞争可能导致野指针、双重释放、use-after-free，进而触发空指针访问或堆破坏。 |
| **难以复现** | 结果依赖线程调度时机，Debug 模式常因额外调试代码掩盖问题，Release 才偶发。 |
| **dump 定位困难** | 崩溃点往往不是根因，真正原因是之前某次无保护写操作已经把内存改坏。 |

因此，这个 demo 并非为了展示"立刻崩溃"，而是为了演示 **一种会间接导致崩溃、且极难定位的底层问题**，并验证 `std::mutex`、`std::atomic` 等治理策略是否有效。

## 5. 深入分析

### 5.1 为什么多线程 bug 难以复现

数据竞争的结果取决于线程调度时机，而线程调度受以下因素影响：

- CPU 核心数量和负载。
- 操作系统调度策略。
- 当前系统中其他进程的状态。
- 编译器优化级别（Debug / Release）。
- 程序中无关代码的微小改动。

因此：

- 单核 CPU 上可能因时间片轮转而“恰好”不出问题。
- Debug 模式下由于额外调试代码和未优化，竞争窗口可能被掩盖。
- Release 模式下优化更激进，竞争条件反而更容易暴露。

### 5.2 Debug 与 Release 的差异

| 特性 | Debug | Release |
|------|-------|---------|
| 优化 | 关闭或低级别 | 高级别优化 |
| 临界区指令数 | 较多，竞争窗口大 | 较少，竞争窗口小 |
| 复现难度 | 可能更难复现 | 可能更容易复现 |
| 调试信息 | 完整 | 可能缺失 |

注意：Release 竞争窗口小不等于更安全。优化可能把多个操作合并或重排序，产生 Debug 下不会出现的执行路径。

### 5.3 `std::atomic` 的内存顺序

本实验使用的是 `std::atomic` 的默认内存顺序，即 `std::memory_order_seq_cst`（顺序一致性）。它提供最强的同步保证，但在某些高性能场景下可能成为瓶颈。

C++11 还提供以下内存顺序：

- `memory_order_relaxed`：只保证原子性，不保证顺序。
- `memory_order_acquire` / `memory_order_release`：用于生产者-消费者模型。
- `memory_order_acq_rel`：同时具有 acquire 和 release 语义。

在不需要强一致性的场景，合理选择更宽松的内存顺序可以提升性能，但也更容易出错。

### 5.4 锁的粒度与性能

测试 2 中每次自增都加锁：

```cpp
for (int j = 0; j < incrementsPerThread; j++) {
    std::lock_guard<std::mutex> lock(g_mutex);
    g_counterLocked++;
}
```

这种细粒度锁虽然正确，但会频繁竞争锁，导致性能下降。更好的做法是：

- 每个线程先累加到本地变量，最后一次性合并。
- 使用原子变量代替锁。
- 使用读写锁（`std::shared_mutex`）优化读多写少场景。

### 5.5 死锁风险

使用锁时必须注意死锁：

- 避免嵌套加锁时顺序不一致。
- 避免在持有锁时调用可能阻塞的函数。
- 考虑使用 `std::lock` 同时锁住多个互斥量。
- 考虑使用 `std::scoped_lock`（C++17）自动处理多锁顺序。

## 6. 工程启示

1. **共享可变数据必须同步**。只要多个线程可能同时读写同一份数据，就必须使用锁、原子变量或其他同步机制。
2. **优先使用 `std::atomic`**。对于计数器、标志位等简单场景，原子操作比锁更轻量。
3. **使用 RAII 锁管理**。`std::lock_guard`、`std::unique_lock`、`std::scoped_lock` 可以自动释放锁，避免忘记解锁。
4. **减小临界区**。只在临界区内做必要的共享数据操作，把计算、IO 等操作移到临界区外。
5. **避免在临界区内调用可能阻塞的 API**。例如网络请求、文件 IO、等待其他锁等。
6. **使用工具检测数据竞争**。例如 ThreadSanitizer（TSan）、Visual Studio 的并发可视化工具、Intel Inspector 等。
7. **设计阶段消除共享**。尽量使用线程本地存储（TLS）、消息传递、不可变数据等无共享设计，从源头避免竞争。

## 7. 本次编译问题回顾

### 7.1 错误现象

初次编译时报：

```text
warning C4819: 该文件包含不能在当前代码页(936)中表示的字符。请将该文件保存为 Unicode 格式以防止数据丢失
error C3872: “0xe1e4”: 此字符不允许在标识符中使用
error C2001: 常量中有换行符
```

### 7.2 根因

`07_race_condition.cpp` 源文件为 **UTF-8 无 BOM** 编码，而 Visual Studio 默认按 GBK（代码页 936）解析源文件，导致中文字符被错误拆分，编译器把乱码字符当作标识符或字符串的一部分，从而报错。

### 7.3 解决方法

将源文件保存为 **UTF-8 with BOM**。这样 Visual Studio 能正确识别编码，中文字符串和注释都能正常编译。

替代方案：在项目属性中设置 **C/C++ → 命令行 → 附加选项** 添加 `/utf-8`，强制编译器按 UTF-8 解析所有源文件。
