# MSVC 运行时库链接方式总结

## 1. 四个选项的含义

MSVC 中 C/C++ 运行时库（CRT）的链接方式由四个选项控制：

| 选项 | 全称 | CRT 链接方式 | 是否 Debug | 是否依赖外部 DLL |
|------|------|--------------|------------|------------------|
| `/MT` | Multi-threaded | 静态链接 | Release | 否 |
| `/MD` | Multi-threaded DLL | 动态链接 | Release | 是（`vcruntime140.dll`、`msvcp140.dll`） |
| `/MTd` | Multi-threaded Debug | 静态链接 | Debug | 否 |
| `/MDd` | Multi-threaded Debug DLL | 动态链接 | Debug | 是（`vcruntime140d.dll`、`msvcp140d.dll`） |

## 2. MT 与 MD 的核心区别

### `/MT`：静态链接

- CRT 的代码被打包进 EXE 或 DLL 内部。
- 运行时不依赖 `vcruntime140.dll`、`msvcp140.dll` 等外部 DLL。
- **每个模块拥有自己独立的 CRT 堆**。

### `/MD`：动态链接

- CRT 以 DLL 形式存在，多个模块共享同一份 CRT。
- 运行时需要安装对应的 VC++ Redistributable。
- **所有使用 `/MD` 的模块共享同一个 CRT 堆**。

这个差异直接影响跨模块内存分配的安全性。在 `14_cross_module_main.cpp` 中可以看到：如果 DLL 和主程序都使用 `/MD`，跨模块 `new`/`delete` 是安全的；如果混用 `/MT` 和 `/MD`，就会因为堆不匹配导致崩溃。

## 3. d 后缀的含义

- 带 `d` 的是 Debug 版 CRT，包含断言、调试堆、guard bytes、额外检查等。
- 不带 `d` 的是 Release 版 CRT，经过优化，没有调试辅助功能。
- **Debug 和 Release 的 CRT 不能混用**。一个模块用 `/MDd`，另一个模块用 `/MD`，跨模块传递内存同样会出问题。

## 4. 对 crash 学习的影响

| 现象 | 原因 | 与运行库的关系 |
|------|------|----------------|
| 跨模块 `new`/`delete` 崩溃 | 堆不匹配 | `/MT` 每个模块独立堆，`/MD` 共享堆 |
| Debug 下能复现的崩溃 Release 下不崩 | Debug CRT 有更严格的检查 | `/MDd` 有 guard bytes，`/MD` 没有 |
| 程序在别人电脑上跑不起来 | 缺少 VC++ Redistributable | 使用了 `/MD` 但没安装运行库 |
| Debug 下变量值显示为 `0xCCCCCCCC` | Debug CRT 自动填充未初始化内存 | `/MDd` 或 `/MTd` 的特性 |

## 5. 配置建议

| 场景 | 推荐选项 | 说明 |
|------|----------|------|
| 日常 Debug 调试 | `/MDd` | 有完整调试信息，能更快发现内存问题 |
| 正式发布 Release | `/MD` | 体积更小，现代 Windows 程序主流选择 |
| 独立绿色版程序 | `/MT` | 不依赖 VC++ Redistributable，但体积增大 |
| 跨模块内存交互 | 统一 `/MD` 或统一 `/MT` | 避免混用导致堆不匹配崩溃 |

## 6. 检查当前项目配置

在 Visual Studio 中：

1. 右键项目 → **属性**
2. **配置(C)** 选择 Debug 或 Release
3. 进入 **C/C++ → 代码生成 → 运行库**
4. 查看当前使用的是 `/MDd`、`/MTd`、`/MD` 还是 `/MT`

建议保持 Debug 用 `/MDd`、Release 用 `/MD`，这是最常见也最不容易出问题的组合。

## 7. 与本项目的关系

- `01_veh.cpp`、`02_seh.cpp`、`03_uef_minidump.cpp` 等 demo 对运行库没有特殊要求，`/MDd` 即可。
- `14_cross_module_main.cpp` 专门演示跨模块内存分配问题，对 `/MT` 和 `/MD` 的差异最为敏感，学习时应重点对比两种配置下的行为。
